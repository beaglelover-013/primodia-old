# 普利莫迪亚 20-9 最小闭环完整检查

检查时间：2026-05-31

本次只检查“一个最小经营闭环”，不扩展到地图、农田、酒窖、人物羁绊、新闻、战斗或大世界。20-9 的核心标准是：七件基础能力必须能在同一条玩法路线里连续跑通。

## 本次验收路线

建议把普利莫迪亚的第一条地基闭环固定为四个叙事回合：

1. `VISIT_SHOP`：前往某处商铺，前端记录当前位置与商铺货架，AI只负责叙述抵达、店主、环境。
2. `BUY_ITEMS`：购买货架里的物品，前端先扣钱、入库、记录交易，AI只负责叙述购买。
3. `COOK_DISH`：选择库房物品与做法，前端先扣除材料、记录烹饪动作或生成待确认成品，AI只负责叙述烹饪过程与结果氛围。
4. `SERVE_DISH`：选择成品上菜，前端先扣除成品、收钱、记录账单，AI只负责叙述客人反应。

## 验收结论

当前 20-9 不能判定为“完整通过”。它已经通过了“进店 -> 买东西 -> 叙述 -> 入库/扣钱/记录”的半闭环，但还没有通过完整的“买菜 -> 做菜 -> 上菜收钱”闭环。

最准确的状态是：

```text
进店：部分通过
买东西：基本通过
做菜：未通过严格伪0层验收
上菜收钱：未通过严格伪0层验收
刷新续玩：部分通过
```

## 主要缺口

### 1. `COOK_DISH` 还不是脚本裁判

目前做菜会把“用某做法做某些材料”写进行动文本，并增加厨艺经验，但没有由脚本统一完成：

- 检查材料是否足够
- 扣除材料
- 生成待确认烹饪结果
- 把结果写入成品、酒水、桶中熟成或失败记录
- 写入聊天变量快照

这会导致后续库存可能依赖 AI/MVU，出现“这回合看着有，下一回合丢了”的风险。

### 2. `SERVE_DISH` 还不是脚本裁判

目前上菜会写入“将某物上桌，收取多少钱”的行动文本，但没有严格完成：

- 检查是否为可上桌成品或酒水
- 扣除上桌数量
- 增加钱
- 记录账单
- 写入聊天变量快照

所以上菜收钱这一段还不能算闭环稳定。

### 3. `VISIT_SHOP` 还没有完全统一进 dispatcher

固定商铺和自由找店都能工作，但它们仍是页面函数直接改位置和商铺状态，再发叙述请求。短期可以用，长期应该变成统一动作：

```text
VISIT_SHOP -> 校验当前位置/输入意图 -> 写当前位置/商铺状态 -> 请求AI叙述
```

### 4. AI/MVU 仍在参与状态更新

项目已经在保护前端状态，但 `runUnifiedNarrativeRequest()` 仍会解析 MVU/JSONPatch 并尝试合并。短期这是兼容层，长期 20-9 要完全通过，至少主闭环内的库存、钱、地点、商铺状态不能再依赖 AI 来补。

## 20-9 通过标准

只要下面这条路线可以连续跑三次，并且刷新后仍正确，就可以把 20-9 判定为通过：

1. 去任意商铺。
2. 买任意 1-3 个商品。
3. 回到库房，用这些商品做菜。
4. 前端扣除材料，并出现成品或待确认成品。
5. 上菜。
6. 前端扣除成品，增加收入。
7. 刷新页面。
8. 钱、库房、当前位置、最新正文、引擎记录都保持一致。
9. 从读档点继续行动，不会接到错误楼层，也不会恢复错商铺。

## 下一步建议

20-9 的下一步不要再扩新玩法，优先补两件事：

1. 做 `COOK_DISH` 的脚本结算：先只做最简单版本，扣材料并生成一个“待AI命名成品”或“临时成品”。
2. 做 `SERVE_DISH` 的脚本结算：扣成品、收钱、写账单。

这两件事完成后，普利莫迪亚的第一条经营闭环才真正落地。
